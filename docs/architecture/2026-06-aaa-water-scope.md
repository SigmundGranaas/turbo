# AAA Water for the wgpu map — scope

## Goal

Replace the current **normal-mapped flat-fill water** with a **displaced-geometry,
environment-reflecting ocean** that reads as AAA (Sea-of-Thieves / modern-engine
tier) at close, tilted hero views — while still behaving across the map's full
zoom range (metres → thousands of km) and running on a Pixel-class mobile GPU.

This is a real feature (multiple days), not a shader tweak. The current shader
(domain-warped exp-sin normals + Fresnel + SSS + sun glitter + heightfield
terrain reflection) is the *normal-mapped ceiling*; the gap to AAA is **two
things it structurally cannot do**:

1. **Wave geometry** — real 3-D displacement so waves have silhouette, parallax,
   and self-occlusion (not a flat sheet with a normal map).
2. **A real environment reflection** — the sky (with clouds), sun, and terrain
   mirrored with detail, instead of a smooth analytic-sky gradient sampled by a
   per-pixel heightfield march.

## Why our case is harder than a normal ocean demo

Every AAA ocean reference (projected grid, Tessendorf FFT) assumes **one infinite
water plane**. Ours is neither infinite nor single-plane:

- Water is **arbitrary MVT polygons** (lakes, rivers, sea) — bounded, with
  coastlines, not an infinite plane.
- Water sits at **many elevations**: sea at z=0, each lake at its own `elev_m`
  (draped on the DEM). A single mirror plane / projected grid doesn't fit.
- The mesh is **lyon fill triangulation** — minimal large interior triangles, so
  there are *no vertices to displace*.
- Target is a **mobile GPU** already under tile-pipeline pressure, across an
  extreme zoom range.

These three facts drive every decision below.

## The decisions (with recommendations)

### D1 — Displaceable geometry: subdivided polygon grid (NOT projected grid)

The projected grid ([Johanson & Lejdfors 2004](http://habib.wikidot.com/projected-grid-ocean-shader-full-html-version))
is the standard, but it assumes an infinite plane and relies on the z-buffer to
clip against geometry — it can't represent lakes at different elevations or
coastline-bounded bodies cleanly.

**Recommend:** at tessellation time, refine each water polygon into a **uniform
fine grid** (constrained to the polygon, ~1–4 m grid in tile-local space at the
native zoom) so it has vertices to displace. Reuse the existing per-tile water
mesh path; add a grid-refinement step after the lyon fill (or replace the fill
with a marching-grid clip against the polygon). LOD: coarser grid for distant
tiles (drop grid resolution with the tile's screen size). This keeps water
exactly where the MVT says it is, at the right elevation, and gives us vertices.

Risk: grid refinement of complex coastlines is fiddly + adds vertices (memory +
tessellation cost). Mitigate with aggressive distance LOD and a max grid budget
per tile.

### D2 — Waves: Gerstner displacement now, FFT later

- **Phase 1: Gerstner** — sum 4–8 Gerstner waves in the **vertex shader** (real
  xy+z displacement → trochoidal crests, silhouette). Cheap, no new infra, works
  on the subdivided grid. Drives the geometry; the existing domain-warped exp-sin
  becomes the **fine-detail normal** layered on top.
- **Phase 3 (optional, gold standard): FFT/Tessendorf** — a tiling
  displacement+normal+foam set generated by inverse-FFT in a **compute shader**
  (equivalent to thousands of Gerstner waves;
  [Tessendorf](https://rtryan98.github.io/2025/10/04/ocean-rendering-part-1.html)).
  This is **new compute infrastructure** (the engine has zero compute today) —
  scope it only if Gerstner+detail isn't enough.

### D3 — Reflection: SSR (terrain) + sky probe, mip-blurred — NOT planar

[Planar reflection](https://dev.epicgames.com/documentation/en-us/unreal-engine/planar-reflections-in-unreal-engine)
is "objectively superior at scale" but **re-renders the scene per plane** — one
plane only, expensive, and we have many lake planes. Not viable here.

**Recommend SSR** (cheaper, all-platform, on-screen only — leaks off-screen):
- Add the missing pieces from the *old* SSR plan: a **linear-depth target** +
  reuse the existing **`hdr_resolve`** (resolved HDR scene colour) as the SSR
  source. Restructure the frame: opaque pass → resolve colour+depth → water pass
  samples them, marches the reflected ray in screen space, **mip-blurred by
  roughness/distance** (the blur is what stops the moiré we already hit).
- **Off-screen / miss → sky**: sample a small **sky+cloud probe** (render the
  analytic sky + the cloud overlay into a tiny cubemap/lat-long once per N
  frames) instead of the flat analytic gradient — this is what gives the
  reflection *content* (the current "plain blue reflection" problem).
- Keep the heightfield march only as a far-distance fallback.

### D4 — Material: refraction + absorption + foam

With colour+depth available (D3), add the rest of the AAA water material:
- **Refraction + depth absorption**: sample the scene colour behind the surface,
  Beer's-law extinction by water depth (shallow→clear, deep→dark) — the "see the
  lake bed near shore" look.
- **Foam**: shoreline (depth-edge) + wave-crest (Jacobian/whitecap) + flow.
- Keep SSS, sun glitter, Fresnel — retuned against the new reflection.

### D5 — Zoom/LOD + mobile budget

- Geometry grid resolution + Gerstner octave count + SSR step count all scale
  with the tile's on-screen size (analytic distance LOD — **not** `fwidth`, which
  we proved is unreliable across the device GPU vs desktop Metal).
- Hard frame-time budget; the device is already tile-pipeline-bound. The reflection
  re-structure (extra resolve + SSR) is the main cost — measure on-device early.

## Phased plan

| Phase | Deliverable | AAA elements gained | Rough effort |
|------|-------------|---------------------|--------------|
| **0** | Frame restructure: linear-depth target + reuse `hdr_resolve`; water moves to its own pass reading them. Subdivided water-grid mesh (D1) behind a flag. | Foundation (no visible change) | 1–1.5 d |
| **1** | Gerstner vertex displacement (D2) on the grid + existing detail normals. | Real 3-D wave shape, silhouette, parallax | 1–1.5 d |
| **2** | SSR terrain reflection + sky/cloud probe, mip-blurred (D3). | Rich, detailed reflection (mountains + clouds + sun) | 2 d |
| **3** | Refraction + depth absorption + foam (D4); retune material. | Shoreline clarity, foam, depth colour | 1.5 d |
| **4** | LOD + mobile perf pass (D5); on-device validation; flag default-on. | Ships at frame budget across zooms | 1 d |
| **5 (opt)** | FFT/Tessendorf compute ocean replacing Gerstner. | Hero-tier wave field | 2–3 d |

≈ **7–8 days** for Phases 0–4 (the AAA jump); +2–3 for the optional FFT tier.

## Validation discipline (lesson from the normal-mapped attempt)

- Render **locally in the golden at the actual close/tilted cameras AND a far
  camera** and *look* before every device ship — the desktop Metal golden hid
  both the moiré and the corduroy that only showed on the device GPU.
- No `fwidth`-based LOD (driver-divergent). Analytic distance LOD only.
- Keep the existing GPU golden green at each phase; add close-up + far preview
  dumps to `target/water-smoke/`.

## Reuse (don't reinvent)

- The water vertex path already drapes on the DEM and is bound to the vector
  pipeline's camera/tile groups — extend it, don't replace.
- `hdr_resolve` already exists as a resolved HDR colour target (SSR source).
- The cast-shadow/AO heightfield (group 3 binding 1) for the far reflection
  fallback + shoreline depth.
- The cloud overlay for the sky probe content.
- `WaterConditions::from_forecast` (MET wave/wind) already drives wave direction
  + ferocity — feed Gerstner amplitude/steepness/direction from it.
